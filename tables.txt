================================================================================
VisiChek Tables — Backend Spec for List Routes, Filters, Pagination & Bulk Actions
================================================================================

Audience: backend engineers
Author:   frontend
Date:     2026-05-10
Status:   open — frontend currently works around several gaps client-side

This file is the authoritative request from the frontend for what every list
endpoint should support so that tables, tabs, filters, pagination and bulk
actions can be wired correctly. Items marked [BLOCKING] are causing visible
bugs or perf issues today. Items marked [WORKAROUND] are things the frontend
is currently faking client-side.


================================================================================
0. CROSS-CUTTING REQUIREMENTS (read first)
================================================================================

0.1  PAGINATION CONTRACT  [BLOCKING]
─────────────────────────────────────
Today /v1/tenants and several other list endpoints return up to 100 rows by
default and the frontend can't tell whether more exist. Tables that genuinely
hold >100 rows (tenants, audit logs, visitor sessions) silently truncate.

REQUIRED for every list endpoint:

  • Accept `skip` (int, default 0) and `limit` (int, default 25, max 200).
    Settle on `skip`/`limit` everywhere; deprecate the older `start`/`stop`
    pair on /v1/visitors/sessions/*. (CLAUDE.md currently documents both —
    we want to converge on one.)
  • Return responses in this envelope:

        {
          "data": [...],
          "meta": {
            "total":    1234,   // total rows matching the filter set
            "skip":     0,
            "limit":    25,
            "hasMore":  true
          }
        }

    Without `total`, the frontend cannot render "Page X of Y" for server-side
    pagination. We are willing to accept `hasMore` only on extremely hot
    endpoints (visitor live feed) where COUNT(*) is too expensive — flag
    those explicitly so we render an infinite-scroll UI instead.
  • Every list endpoint must accept `?limit=` so a small front-end tool can
    bump it for debugging, with the hard cap enforced server-side.

0.2  STABLE SORT  [BLOCKING]
─────────────────────────────
Pagination without a stable order is meaningless. Every list endpoint must:

  • Accept `sort` query param: comma-separated `field` or `-field` (descending),
    e.g. `?sort=-dateCreated,companyName`.
  • Have a documented default sort that includes a tiebreaker on `id` so two
    rows with the same `dateCreated` always resolve to the same page.
  • Return 400 with `data.code = INVALID_SORT_FIELD` if the client passes a
    sort field the endpoint does not allowlist.

0.3  SEARCH PARAM  [BLOCKING]
──────────────────────────────
Replace the per-endpoint `?q`/`?search`/`?email` zoo with a single, consistent:

  • `?q=<string>` — case-insensitive partial match against a documented set
    of fields per resource (listed below).
  • Trim whitespace, ignore queries shorter than 2 chars (return everything).
  • Combine with filters using AND, not OR.

0.4  FILTER NAMING  [WORKAROUND]
─────────────────────────────────
Filters should be plain query params with these conventions:

  • Single value:        `?status=active`
  • Multiple values:     `?status=active&status=trialing` (repeated key) OR
                         `?status=active,trialing` (CSV) — pick one and stick
                         to it across all endpoints. The frontend strongly
                         prefers repeated key (RFC 6570 form-style).
  • Date ranges:         `?createdAtGte=<unix>&createdAtLte=<unix>`
                         (always unix epoch seconds; consistent with the rest
                         of the API per CLAUDE.md).
  • Boolean:             `?isActive=true` (literal "true"/"false", not 1/0).
  • Null/empty:          `?planTier=none` for "no plan attached".

0.5  BULK ACTION ENDPOINTS  [WORKAROUND]
─────────────────────────────────────────
The frontend currently fans out N single-item mutations via Promise.allSettled
for every bulk action. This is wasteful (N round-trips, N audit-log entries
per action, N invalidations of the same query). For each mutation listed in
the per-resource sections, we want a sibling bulk endpoint:

    POST /v1/<resource>/bulk/<action>
    body: { ids: string[], <action-specific extras> }
    response:
      {
        "succeeded":  ["id1", "id2"],
        "failed":     [{ "id": "id3", "error": { "code": "...", "message": "..." } }]
      }

The endpoint MUST be all-or-nothing optional via a `?atomic=true` flag (default
false). When atomic, any failure rolls back the whole batch and returns 422
with the same shape so the UI can highlight the failing rows.

Audit logs: emit ONE bulk audit entry per bulk call (with the id list inside),
not N entries. This is what auditor expects.

0.6  CSV / EXPORT  [NICE-TO-HAVE]
──────────────────────────────────
Several read-only views (audit logs, billing invoices, payments) want a
"Download selected" or "Download all matching the current filter" action.
Add `Accept: text/csv` on the same list endpoints, with the same filter
params, returning a streaming CSV. Same auth and pagination rules apply
(but `limit` should default to 10k for CSV, capped at 100k).

4. ADMIN-WIDE NICE-TO-HAVES
================================================================================

  • Saved filters per user.   Each list endpoint persists named filter sets
    keyed on user_id + resource. POST /v1/me/saved-filters and
    GET /v1/me/saved-filters?resource=tenants. Frontend will surface them
    as a "Saved views" dropdown next to the search box.

  • Column preferences.       Same shape but for which columns are visible
    and their order. POST /v1/me/column-prefs?resource=tenants.

  • Webhook notifications.    For long-running bulk actions (>5s) the
    frontend wants either:
      – an HTTP 202 with `{ jobId }` and `GET /v1/jobs/{jobId}` polling
        (matches the existing in-flight-jobs hook), or
      – a websocket / SSE stream on /v1/jobs/{jobId}/events.

  • Idempotency keys.         All bulk POSTs should accept
    `Idempotency-Key: <uuid>` and replay the cached response on retries
    within 24h. Critical for delete/offboard so a flaky network doesn't
    double-process.
