===============================================================================
 VISICHEK INSIGHTS — BACKEND DATA REQUIREMENTS
 What the frontend needs from the backend to power the role-tabbed Insights
 page, replacing the current tenant dashboard.
===============================================================================

Author:  frontend
Status:  request to backend
Scope:   tenant shell only (super_admin, dept_admin, receptionist, auditor,
         security_officer, dpo). The platform-admin shell keeps its own
         /v1/admins/dashboard/stats contract and is OUT OF SCOPE here.

-------------------------------------------------------------------------------
0. CONTEXT — WHAT THIS REPLACES
-------------------------------------------------------------------------------

Today the tenant dashboard reads ONE fat payload from:

    GET /v1/dashboard/stats            -> TenantDashboardStats
    (see src/types/dashboard.ts for the full existing shape)

That payload is good but it is:
  - fixed-window only (it hard-codes "last 7 days" and "last 30 days"
    series; there is no caller-chosen date range),
  - role-blind (it returns everything to everyone and the client picks),
  - plan-blind (it does not describe what the current plan is allowed to see).

The new Insights page is the replacement design. It is organised as:

    [ Role selector ]  ->  picks WHOSE lens (defaults to the signed-in role)
    [ Time range ]     ->  adjustable window (NOT just 7/30 — see section 2)
    [ Per-role tabs ]  ->  Overview + 2 role-specific tabs
    [ Section grid ]   ->  KPIs + charts that recompute when range/role change

When this ships it becomes the tenant landing dashboard. The old fixed
dashboard layout is retired. So the contract below must satisfy BOTH the
"glance" dashboard need and the "explore" analytics need in one place.

-------------------------------------------------------------------------------
1. GLOBAL CONVENTIONS (already true on the platform — restating so the new
   endpoint follows them)
-------------------------------------------------------------------------------

- Response envelope: { success, message, data, meta?, requestId? }.
  Everything below describes the UNWRAPPED `data`.
- Field casing: camelCase by default (CaseConversionMiddleware). Honour
  X-Response-Case: snake if sent.
- Timestamps: unix EPOCH SECONDS, UTC. Never ISO except health probes.
- Pagination (where lists are paged): skip / limit. (start/stop is dead —
  the backend now 400s on start/stop.)
- Plan tier is exposed on tenant routes via X-Plan-Tier and
  X-Subscription-Status response headers, AND via the limitations payload
  the frontend already consumes (plan.tier, deniedFeatures[], caps{},
  plan.isFreeFallback). Analytics gating MUST reuse that mechanism — do not
  invent a parallel one (see section 7).
- All numeric metrics default to 0 and all lists default to [] for brand-new
  tenants, so every section is safe to render unconditionally.

-------------------------------------------------------------------------------
2. THE TIME RANGE MODEL  (most important new requirement)
-------------------------------------------------------------------------------

The time range must be MASSIVELY adjustable, not a 7/30/90 toggle. The user
can pick any custom window. BUT the window is bounded by how long the tenant
has actually existed on the platform — you cannot scroll back to before the
tenant was created (there is no data there and the picker must not allow it).

2.1 REQUEST PARAMS (new — added to the analytics endpoint, section 3)

    start        unix seconds, inclusive lower bound of the window
    stop         unix seconds, inclusive upper bound (defaults to "now")
    granularity  optional override: "hour" | "day" | "week" | "month"
    role_view    one of the 6 SystemUserRole values; defaults to caller role
    department_id optional dept scope (see per-role rules in section 6)

    Presets the UI will offer (all just map to a start/stop pair, the server
    does not need to know preset names): Today, 7d, 30d, 90d, This month,
    This quarter, This year, Custom (date-range picker), and "All time"
    (= start at tenantCreatedAt).

2.2 TENURE BOUND (new — server must enforce + advertise)

    - The response meta MUST include:
          tenantCreatedAt   unix seconds — when this tenant was created
          earliestData      unix seconds — first day we actually have data
                            (>= tenantCreatedAt; lets the picker grey out
                             empty leading days)
      The frontend uses tenantCreatedAt to set the minimum selectable date
      in the range picker. "All time" resolves to this value.
    - If `start` is earlier than tenantCreatedAt, CLAMP it to tenantCreatedAt
      (do not error). Echo the effective start/stop back in meta.appliedRange
      so the UI can show "showing 14 May – 22 May (clamped to account start)".
    - If `stop` < `start` after clamping -> 422 VALIDATION_ERROR.

2.3 AUTO-GRANULARITY (server picks bucket size from window length unless the
    caller overrides via `granularity`)

        window <= 48h           -> hourly buckets   (24-48 points)
        window <= 31 days       -> daily buckets
        window <= 180 days      -> weekly buckets
        window  > 180 days      -> monthly buckets

    Rationale: a 1-year line at daily granularity is 365 noisy points; weekly/
    monthly keeps every series readable and the payload small. Always return
    the granularity actually used in meta.granularity, and ZERO-FILL gaps so
    the series is continuous (no missing buckets).

2.4 TIME SERIES POINT SHAPE (reuse existing TimeSeriesPoint)

        { timestamp: number,   // unix seconds at bucket start, UTC
          label: string,       // "YYYY-MM-DD" for day/week/month,
                               //  "HH:00" for hour buckets
          value: number }

    Every series in every section uses this shape so the same chart
    component renders all of them.

-------------------------------------------------------------------------------
3. THE ENDPOINT  (PROPOSED — extends the existing dashboard stats route)
-------------------------------------------------------------------------------

    GET /v1/dashboard/insights
        ?role_view=<role>
        &start=<unix>&stop=<unix>
        &granularity=<hour|day|week|month>   (optional)
        &department_id=<id>                  (optional, see section 6)
        &branch_id=<id>                      (optional, super_admin/recept.)

    -> data: InsightsResponse  (shape below)

We are OK with the backend instead extending GET /v1/dashboard/stats to take
start/stop/role_view/granularity and return the role-scoped+ranged shape —
whichever is cleaner server-side. Either way the response must be ROLE-SCOPED
(only compute the sections that role's tabs need; section 6) and RANGE-AWARE.

    InsightsResponse {
      meta: {
        roleView:       SystemUserRole
        departmentId:   string | null
        branchId:       string | null
        tenantCreatedAt: number          // for the range-picker lower bound
        earliestData:   number
        appliedRange:   { start: number; stop: number }   // post-clamp
        granularity:    "hour" | "day" | "week" | "month"
        planTier:       string           // echo, drives gating fallback
        availableSections: string[]      // section keys this plan may see
        lockedSections:    string[]      // section keys gated behind upgrade
        lastUpdated:    number
      }
      kpis:        Kpi[]                  // role-specific, section 4
      sections:    Record<string, Section>  // keyed by section id, section 5
    }

Caching: keep the 60s precompute TTL behaviour you already have for the
unscoped view. Custom ranges that are not "now-anchored" (i.e. a fixed
historical window) are immutable and can be cached hard. Department-scoped
and custom-range requests may bypass the precompute cache and compute on
demand — that is acceptable.

-------------------------------------------------------------------------------
4. KPI CARDS  (the "kpis" array — 4 per role)
-------------------------------------------------------------------------------

Each KPI is:

    Kpi {
      key:        string          // stable id, e.g. "totalVisits"
      label:      string          // "Total visits"
      value:      number | string // already formatted-friendly (raw number
                                  //  preferred; client formats)
      unit:       string | null   // "%", "min", null
      trend: {                    // optional comparison vs the PRECEDING
        changePercent: number     // window of equal length
        direction: "up" | "down" | "flat"
        isGood: boolean           // up isn't always good (denials up = bad)
      } | null
      description: string         // plain-language sub-label
    }

IMPORTANT: trend compares the selected window against the immediately
preceding window of the SAME length (e.g. selected = last 14 days -> compare
to the 14 days before that). Not a fixed WoW. `isGood` lets the UI colour the
arrow correctly per metric.

KPI sets the frontend will render per role (give us these keys/values):

  super_admin      totalVisits, currentlyActive, verificationRate (%),
                   openIncidents (+ criticalIncidents in description)
  dept_admin       deptVisitors (in range), appointmentsInRange,
                   noShowRate (%), activeHosts
  receptionist     checkedInToday, awaitingCheckout, avgWaitMinutes (min),
                   badgesPrinted (in range)
  auditor          auditEventsToday, auditEventsInRange, uniqueActors,
                   exportsInRange
  security_officer openIncidents, criticalIncidents, approachingDeadline
                   (NDPC 72h), resolvedInRange
  dpo              openDsr, consentRate (%), retentionPolicies, subProcessors

Most of these already exist as scalars on TenantDashboardStats; the new bit
is that the "in range / in window" ones must respect start/stop, and each
needs the trend object computed against the preceding window.

-------------------------------------------------------------------------------
5. SHARED SECTION SHAPES  (the chart building blocks)
-------------------------------------------------------------------------------

The Insights UI composes each tab from a small set of section types. Every
section in `sections` is one of:

  TIME_SERIES  -> { type:"timeSeries", title, points: TimeSeriesPoint[],
                    valueLabel }                         // line/area chart
  DISTRIBUTION -> { type:"distribution", title,
                    slices: DistributionSlice[] }        // pie/donut
  HOURLY       -> { type:"hourly", title, buckets: HourlyBucket[] }  // bar strip
  TOP_LIST     -> { type:"topList", title, items: TopItem[] }       // ranked bars
  FEED         -> { type:"feed", title, events: RecentCheckIn[] }    // live list

(DistributionSlice, HourlyBucket, TopItem, RecentCheckIn already exist in
src/types/dashboard.ts — reuse them verbatim.)

The concrete section ids the frontend asks for, and their data source:

  id "traffic"        TIME_SERIES  visit check-ins per bucket over the range
  id "audit"          TIME_SERIES  audit events per bucket over the range
  id "hourly"         HOURLY       check-ins by hour-of-day, aggregated over
                                   the WHOLE selected range (0-23, zero-filled)
  id "visitStatus"    DISTRIBUTION visit sessions by VisitStatus in range
  id "incident"       DISTRIBUTION incidents by IncidentStatus in range
  id "dsr"            DISTRIBUTION DSRs by DSRType in range
  id "appointment"    DISTRIBUTION appointments by AppointmentStatus in range
  id "newReturning"   DISTRIBUTION new vs returning visitors in range
  id "topDepartments" TOP_LIST     departments by visit count in range (top 10)
  id "feed"           FEED         most recent check-ins (cap 10; receptionist)

All distributions: omit zero-value slices server-side (already the rule),
percentages rounded to 1 dp summing to ~100.
All "in range" sections: filtered to meta.appliedRange, scoped per section 6.

-------------------------------------------------------------------------------
6. PER-ROLE REQUIREMENTS  (tabs, sections, filters, scoping)
-------------------------------------------------------------------------------

For each role: which tabs exist, which section ids each tab needs (so you only
compute those), the additional FILTERS that role's UI exposes, and the data
SCOPING the backend must enforce regardless of what the client asks for.

........................................................................
6.1 SUPER_ADMIN  — org-wide, every branch & department
........................................................................
Scope:  no implicit restriction; sees all branches/departments of the tenant.
Filters the UI sends:  branch_id (optional), department_id (optional),
                       start/stop, granularity.
Tabs & sections:
  Overview      kpis, traffic, newReturning, topDepartments, hourly, visitStatus
  Compliance    incident, dsr, audit
  Departments   topDepartments, hourly, appointment
Notes: topDepartments should honour branch_id when set. Compliance tab pulls
incident/dsr distributions across the whole tenant.

........................................................................
6.2 DEPT_ADMIN  — scoped to the admin's own department
........................................................................
Scope:  HARD-SCOPED to the caller's department_id. Even if the client omits
        or forges department_id, the server must restrict to the dept(s) this
        admin owns. Cross-department data must never leak.
Filters the UI sends:  host_id (optional, hosts in their dept), start/stop,
                       granularity.  (No department picker — they have one.)
Tabs & sections:
  Overview      kpis, traffic, appointment, topDepartments, hourly, visitStatus
  Appointments  appointment, newReturning, traffic
  Visitors      hourly, visitStatus, topDepartments
Notes: "topDepartments" for a dept_admin degenerates to top hosts/sub-units
within their department — if you cannot split sub-units, return top hosts as
the TOP_LIST items instead and we will relabel client-side; tell us which.

........................................................................
6.3 RECEPTIONIST  — live front desk, their branch
........................................................................
Scope:  the receptionist's branch. Live, near-real-time bias.
Filters the UI sends:  status filter on the feed (VisitStatus subset),
                       start/stop (mostly "today"), granularity.
Tabs & sections:
  Overview      kpis, traffic, hourly, visitStatus, feed
  Live desk     feed, visitStatus, hourly
  Hourly flow   hourly (tall), traffic
Notes: the "feed" section should be the most recent check-ins for the branch,
cap 10, newest first, including status + verified flag + host. The UI polls
this tab faster (the existing 5s active-visitor cadence). Everything else can
use the standard 30-60s staleness.

........................................................................
6.4 AUDITOR  — read-only activity trail
........................................................................
Scope:  whole tenant, READ-ONLY. No write/action sections.
Filters the UI sends:  actor_id (system user), operation_type (OperationType:
                       create/read/update/delete), resource_type (optional),
                       start/stop, granularity.
Tabs & sections:
  Overview      kpis, audit, hourly, visitStatus
  Activity      audit, hourly, topDepartments
  Coverage      visitStatus, incident, topDepartments
Notes: the "audit" TIME_SERIES must respect actor_id / operation_type /
resource_type filters when present. uniqueActors KPI = distinct actors in the
window. exportsInRange KPI = count of compliance/audit export events in window.

........................................................................
6.5 SECURITY_OFFICER  — incident posture & NDPC deadlines
........................................................................
Scope:  whole tenant incidents + supporting visit context.
Filters the UI sends:  incident_type (IncidentType), incident_status
                       (IncidentStatus), severity (if you track it),
                       start/stop, granularity.
Tabs & sections:
  Overview        kpis, incident, visitStatus, traffic
  Incidents       incident, dsr, audit
  Activity trail  audit, hourly, topDepartments
Notes: approachingDeadline KPI = incidents whose notification_deadline
(date_created + 72h, NDPA s.38) is within 24h and not yet ndpc_notified. We
also want, in the incident distribution's meta or a sibling field, the count
of incidents PAST deadline so the UI can render the red banner.

........................................................................
6.6 DPO  — data-subject requests, consent, register
........................................................................
Scope:  whole tenant privacy/compliance data.
Filters the UI sends:  dsr_type (DSRType: access/correction/deletion/
                       consent_withdrawal), dsr_status (DSRStatus), lawful_basis
                       (LawfulBasis), start/stop, granularity.
Tabs & sections:
  Overview   kpis, dsr, visitStatus, traffic
  Requests   dsr, newReturning, audit
  Consent    newReturning, visitStatus, topDepartments
Notes: consentRate KPI = granted / (granted + withdrawn + not-granted) in the
window. We also want consentWithdrawalCount in the window for the Consent tab
sub-label. retentionPolicies / subProcessors are current counts, not windowed.

-------------------------------------------------------------------------------
7. FREE-TIER ANALYTICS  (the LEAST amount of data — explicit, deliberate)
-------------------------------------------------------------------------------

Free tenants (plan.tier == "free" OR plan.isFreeFallback == true) get a
deliberately minimal Insights experience. The goal is "useful enough to prove
value, restricted enough to drive an upgrade."

7.1 WHAT FREE GETS

  - ONLY the Overview tab, for their own role. The 2 extra role tabs are
    rendered as LOCKED (greyed, padlock, upsell tooltip) — not hidden, so the
    user can see what they are missing. The role selector still works but the
    other role's deep tabs are locked the same way.
  - KPIs: only the 4 Overview KPI cards, and WITHOUT the trend object
    (trend = null on free). No window-over-window comparison on free.
  - Time range: LOCKED to a fixed "Last 7 days", daily granularity. The range
    picker is visible but disabled with an upgrade tooltip. No custom range,
    no "all time", no quarter/year. start/stop on the request are ignored for
    free and forced to last-7d server-side.
  - Sections on the Overview tab for free, regardless of role:
        traffic        (7-day daily line, value only — no trend overlay)
        visitStatus    (single distribution donut)
    That is it. Two charts.
  - topDepartments on free: capped at TOP 3 (not 10).
  - hourly, audit, incident, dsr, appointment, newReturning, feed:
    ALL LOCKED on free. Return them in meta.lockedSections, do NOT compute
    or include their data. The UI renders a locked placeholder card with copy
    like "Hourly traffic patterns are available on paid plans" + an Upgrade CTA.
  - No CSV / PDF export of any chart on free.
  - lastUpdated / freshness: free can have a longer cache (e.g. 5 min) since
    they cannot pick ranges anyway.

7.2 HOW FREE IS ENFORCED (must be server-authoritative)

  - The endpoint reads the tenant's plan and populates:
        meta.planTier
        meta.availableSections   // e.g. ["traffic","visitStatus","topDepartments"]
        meta.lockedSections      // everything else this role's tabs reference
    The frontend uses these to decide what to render vs lock. It MUST be
    server-driven so a client cannot just request a locked section and get
    data — locked sections return NO data (omit the key from `sections`, list
    it in lockedSections).
  - This must align with the existing limitations payload the client already
    fetches (deniedFeatures[], caps{}). Add analytics capability keys there so
    nav/menus can pre-lock without a round-trip, e.g.:
        analytics.customRange
        analytics.roleTabs
        analytics.hourly
        analytics.compliance        (incident + dsr sections)
        analytics.audit
        analytics.export
        analytics.trends            (KPI window-over-window)
    deniedFeatures lists the ones this plan cannot use; the Insights UI greys
    those controls/cards and shows the upgrade tooltip.
  - caps{} can optionally carry numeric ceilings we will respect client-side:
        caps.analyticsMaxRangeDays   // free = 7; paid tiers larger/unbounded
        caps.topListMax              // free = 3; paid = 10
    If a cap is absent, treat as unlimited.

7.3 PAID TIERS (for contrast — not exhaustive, backend owns the matrix)

  - Custom range up to caps.analyticsMaxRangeDays (or unbounded), all
    granularities, all presets including "All time" (bounded by tenure).
  - All 3 role tabs unlocked.
  - All section types unlocked (hourly, audit, compliance, feed, etc.).
  - KPI trend comparisons enabled.
  - Export enabled (see section 8).
  The exact tier->feature matrix is the backend/billing team's call; the
  frontend only consumes availableSections/lockedSections + deniedFeatures.

-------------------------------------------------------------------------------
8. EXPORT  (paid only — nice-to-have, flag if not feasible now)
-------------------------------------------------------------------------------

  GET /v1/dashboard/insights/export
      ?format=csv|pdf & <same range/role/filter params as section 3>
  -> StreamingResponse (CSV of the underlying rows, or a rendered PDF of the
     current tab). Gated by analytics.export. Free returns 403 with
     SUBSCRIPTION_REQUIRED + plan context so the UI shows the upgrade prompt.

-------------------------------------------------------------------------------
9. SUMMARY OF WHAT IS NEW VS ALREADY EXISTS
-------------------------------------------------------------------------------

ALREADY EXISTS (reuse as-is):
  - GET /v1/dashboard/stats + TenantDashboardStats (most scalar KPIs, the
    distributions, hourly/day-of-week buckets, top lists, recent check-ins).
  - DistributionSlice, TimeSeriesPoint, HourlyBucket, TopItem, RecentCheckIn,
    GrowthMetric, DashboardPeriod types.
  - The limitations payload (plan.tier, deniedFeatures, caps, isFreeFallback)
    and X-Plan-Tier / X-Subscription-Status headers.

NEW / NEEDED FROM BACKEND:
  1. A range-aware, role-scoped analytics endpoint (extend /dashboard/stats or
     add /dashboard/insights) accepting start/stop/granularity/role_view/
     department_id/branch_id (sections 2 & 3).
  2. Auto-granularity bucketing with zero-fill (section 2.3).
  3. Tenure bound: tenantCreatedAt + earliestData in meta, server-side clamp
     of start to tenantCreatedAt, appliedRange echo (section 2.2).
  4. KPI trend objects compared against the preceding equal-length window,
     with an isGood flag per metric (section 4).
  5. Per-role section scoping + hard data scoping (dept_admin to their dept,
     receptionist to their branch, auditor read-only) (section 6).
  6. Filter support per role: branch_id, host_id, actor_id, operation_type,
     resource_type, incident_type/status, dsr_type/status, lawful_basis
     (section 6).
  7. Server-authoritative plan gating: availableSections / lockedSections in
     meta, plus analytics.* capability keys + analyticsMaxRangeDays /
     topListMax caps in the limitations payload (section 7).
  8. "Past NDPC deadline" count for the security_officer banner (section 6.5).
  9. (Optional) export endpoint, paid-gated (section 8).
